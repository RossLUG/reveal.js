MQTT

RossLUG April 2014

Jon Archer

What is it?

  • MQTT (Message Queue Telemetry Transport) is a lightweight messaging protocol which sits on top of TCP/IP.
  • Extremely Simple and Lightweight.
  • Designed for Constrained devices, with low resources available.
  • Designed for Low Bandwidth or unreliable networks.

    • How?

  • Works on the Publish/Subscribe methodology.
  • Involves a broker which both publisher and subscriber connect to.
  • Utilises topics to distinguish messages.
  • Allows Quality of Service - QoS.
  • Allows the retention of messages.
  • Allows security of connection using SSL

    • Publish/Subscribe?

    Is exactly as it sounds, one side connects to the broker and publishes a message - up to 256mb in size - to a particular topic.


    A single or multiple device also connects to the broker as a subscriber and listens for messages on the particular topic.

    Topics

    Messages are published to specific topics to allow them to be categorised to specific subscribers, a broker could be handling tens of thousands of messages a second for a mass of devices.

    /location/device/node

    /location/device/#

    /location/+/node

    Wildcards and delimiting

    When subscribing to MQTT topics, wildcards (# or +) can also be used. When publishing, a delimeter (/) may be used to create a hierachy within the topic.

    /location/device/node

    /location/device/#

    /location/+/node

    Why?

    With the advent of IoT/M2M there is a need for devices to be able to speak to one another, MQTT offers a really simple way for this to happen, eg. washing machine to tweet when it is finished.

    Where is it used?

  • Facebook Messenger - Presence information
  • FloodNet - Southampton University
  • Mobile4D - Disaster Reporting & Alerting project at University of Bremen
  • A lot of IBM applications

    • Lets take a look

    Looking at a simple home automation device, we could use the topic

  • house/room1/1

    • This could represent the first light in the first room of a house, having an arduino as a subscriber could listen for a particular message on that topic to turn a light on or off.


    What happened there

    The buttons were using a php mqtt library to connect to the broker

    The on button publishes the message 2-40 to topic house/room1/1

    The off button publishes the message 2-41 to topic house/room1/1

    The Arduino subscribes to house/room1/1 and depending upon the message it recieves will turn the lamp on or off

    That was fairly simple??

    Messages could have a fairly complex payload, images, JSON strings. A lot of projects are now either using or considering MQTT as the API for their application.

  • Owntracks - uses JSON strings over MQTT to report GPS data
    • {"_type": "location", "lat": "53.6897209", "lon": "-2.2991612", "tst": "1396091441", "acc": "10.0", "batt": "85"}
  • Android/IOS applications using MQTT for API services
  • Web applications using MQTT over websockets for extra functionality
